Implementation plan

Personas, SEO, speed, analytics, and monetization guardrails

NH-Data is a public-data utility, not a generic blog. The operating plan keeps pages useful, crawlable, fast, and AdSense-safe while avoiding unsupported medical or legal claims.

Site persona

  • Primary reader: US adult-child caregiver comparing nursing homes under time pressure.
  • Promise: clear CMS data, local context, neutral enforcement records, and correction paths.
  • Voice: calm, factual, concise; never sensationalize abuse, fines, or deficiencies.
  • Trust standard: source, date, method, and reviewer visible on every high-stakes page.

Article persona

  • Main job: explain one CMS concept so a reader can use the tool better.
  • Format: short answer, definitions, real table, source stamp, related tool CTA.
  • Evidence: CMS or primary source first; current facts must be verified before publishing.
  • Forbidden: AI filler, keyword stuffing, invented expert claims, fake urgency.

Performance

  • Keep listing/filter interactions client-light and dependency-free.
  • Reserve stable dimensions for cards, badges, comparison rows, and mobile nav.
  • Use source-dated text and real HTML tables rather than image-rendered data.
  • Do not add ad slots; Auto Ads will decide placement after AdSense approval.

SEO

  • Index only pages with unique facility, location, comparison, or guide value.
  • Use one H1, descriptive titles, meta descriptions, canonical tags, and schema where factual.
  • Gate thin filter pages by search demand and minimum result count.
  • Use `/corrections` and `/methodology` as E-E-A-T anchors.

GSC / GA4

  • GSC check needs the final verified property URL or service-account access.
  • GA4 check needs the property ID or token mapping for this domain.
  • `D:\env` contains Google-related tokens, but no NH-Data domain mapping was present in this repo.
  • After launch, verify sitemap coverage, query intent, engagement, and page templates with high exits.

AdSense

  • Auto Ads only. No manual slots are placed in the implementation.
  • Delay AdSense review until methodology, corrections, and reviewer pages are public.
  • Keep enforcement language neutral and source-based to avoid policy and trust risk.
  • Optimize for readable utility pages first; RPM follows intent quality.

Build environment notes

  • Current workspace is a static implementation extracted from the design bundle; it is not currently a Git repository.
  • No Vercel, Turso, or GitHub mutation was performed. Deployment remains limited to future GitHub push only unless explicitly re-authorized.
  • Content in this implementation was written directly in this Codex session; no external writing API was used.
